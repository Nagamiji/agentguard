# Engineering report — 2026-07-17 (cycle 9: Phase 6.1, demo experience & report)

**The cycle that made the wedge something a person can run and understand.**

## 1. The claim

`make demo` — one command — runs the whole product loop and ends with:

```
decision:    BLOCKED
[CRITICAL] tool_arg_limit: 'issue_refund.amount' was 9000, above the permitted maximum of 100
exit code: 20
HTML report: agentguard-demo-report.html   (self-contained, 🔴 BLOCKED, evidence + remediation)
```

A first customer installs, runs one command, and *sees* a blocked deployment with evidence and
a fix — the thing that makes the product believable.

## 2. Architecture review (before building)

The steer was right: **don't build a dashboard first.** The report *is* the minimal
visualization, at a fraction of the surface and none of the new auth/session risk. The
architecture review's conclusion drove the scope: reuse the existing gate/risk/policy endpoints;
add a client-side renderer + a demo; audit production-readiness rather than rewrite.

## 3. What changed

PR (open): `src/agentguard_cli/report.py` (structured report + self-contained HTML + JSON +
per-finding remediation), `--html`/`--report-json` on the CLI, `scripts/demo.sh` + `make demo`,
`docs/quickstart.md`, `docs/production-readiness.md`, ADR 0014, and tests (11 report + a report
assertion in the workflow test). No migration; no new server endpoint.

## 4. Key decisions (ADR 0014)

- **A report, not a dashboard.** Self-contained HTML (no external requests — tested), every
  dynamic value HTML-escaped (a finding is arbitrary agent output; tested with a `<script>`
  payload), remediation per `check_type` so the report says what to *do*.
- **One-command reproducible demo** — register → policy → prompt-injection scenario → scan →
  BLOCKED, deterministic by default, `DEMO_RUNNER=vertex` for a real model, torn down on exit.
- **Production-readiness audit done inline** — full context beats a cold audit agent. It *names*
  the gaps rather than hiding them.

## 5. Production-readiness audit (the honest map)

Solid: tenant isolation (RLS forced + tested per table), auth (hashed 256-bit keys), secrets
(env + ADC + gitleaks), migrations (roundtrip-tested), immutable evidence, no real tool
execution. **Top gaps, named not hidden:** observability (no metrics/tracing/alerting — the
biggest), RBAC / scoped keys, rate limiting / cost controls. Plus the known caveats: advisory
merge gate, symmetric signing, override loosening, project scope, simulation fidelity. None is a
data-safety hole; all are operational / multi-tenant-at-scale. Full table + pilot-vs-GA framing
in `docs/production-readiness.md`.

## 6. Security findings

| Finding | Severity | Status |
|---|---|---|
| A report rendering agent output raw is an XSS vector in the security tool itself | High | All dynamic values HTML-escaped; tested with `<script>` |
| A report fetching external assets could leak the viewer's context / not open offline | Medium | Fully self-contained (inline CSS); tested for no external refs |
| Report enrichment (agent name/policy) failing could crash a valid scan | Medium | Best-effort `_safe` catches `ApiError` → the verdict is authoritative; a broken double surfaced this and was fixed |

## 7. Tests performed

11 report unit tests (structure, remediation coverage, self-contained HTML, XSS escaping,
allowed/blocked rendering) + a report assertion in the workflow test. **A real `make demo`
run**: exit 20, BLOCKED, self-contained 2.8 KB HTML report. Full offline suite **187 passed / 6
skipped**; ruff + `mypy --strict` clean (45 files); gitleaks clean. No migration.

## 8. Risks / deferred

- No hosted/shareable report view (client-rendered file only).
- The audit's gaps (observability, RBAC, rate limiting) are **named, not fixed** — building them
  silently would be the "fake feature" the brief forbids. They are the real next work.
- Earlier limits unchanged.
