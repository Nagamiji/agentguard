# Engineering report — 2026-07-17 (cycle 8: Phase 5, deployment gate)

**The cycle that turned the platform into a CI control a developer feels — a red check.**

## 1. The claim

`agentguard scan`, run as a real subprocess over HTTP against a live server:

```
decision:    BLOCKED
risk:        critical
[CRITICAL] tool_arg_limit: 'issue_refund.amount' was 9000, above the permitted maximum of 100
EXIT CODE: 20        # non-zero → the CI step fails → the merge is blocked
wrote SARIF -> agentguard.sarif   (2.1.0, 1 error-level result)
```

That exit code is the product wedge: it is what stops a merge. The same command on a safe
config exits 0.

## 2. Architecture review (before building)

The server already answered the CI question (`/gate`, `/risk`, policy). So Phase 5 is the
**client + CI surface**, not more platform — which is why it needs **no migration** and adds
only one response field. Reused everything; built the thin layer that was missing.

## 3. What changed

PR (open): `src/agentguard_cli/` (the `agentguard` CLI — scan/evaluate/report/policy check/
fingerprint), `keel/signing.py` + `GateOut.signature`, `.github/actions/agentguard/action.yml`
(composite GitHub Action + SARIF upload), `docs/deployment-gate.md`, ADR 0013, and tests (13
CLI unit + 6 workflow/integration + 3 signing).

## 4. Key decisions (ADR 0013)

- **The exit code is the contract** (`0 allowed · 20 blocked · 10 error · 30 unknown`), fail
  closed by default — a gate that exits 0 when it could not tell is not a gate.
- **Pull model, not webhooks.** The CLI calls the API; no public callback, no webhook-signature
  subsystem this cycle. Deferred honestly.
- **SARIF 2.1.0**, uploaded even on failure, so the developer sees *why* it blocked.
- **Signed verdicts, honestly scoped.** HMAC over `(fingerprint, decision, run_id)` — symmetric
  integrity for a key-holder, **not** third-party non-repudiation (asymmetric is the upgrade).
  Signed over `run_id` not a timestamp, to dodge a serializer mismatch (caught in testing).
- **Reuse over rebuild:** no new tables; the CLI's HTTP layer is a Protocol so the whole flow
  runs in-process against a `TestClient`.

## 5. Security findings

| Finding | Severity | Status |
|---|---|---|
| A CLI that exits 0 on a network/HTTP error would wave a deploy through | High | Every error path → non-zero; tested (unreachable server, HTTP error) |
| A signed verdict that can't be reproduced by the verifier is useless | Medium | Signed over `run_id`, not a datetime; verify tested tamper-evident |
| SARIF echoing a real leaked secret into the PR | Medium | Detail names the pattern; library markers are synthetic; tested |
| CLI missing from the production image (a past class of bug) | Medium | `agentguard` entry point verified in the Docker image |

## 6. Tests performed

13 CLI unit (fingerprint, SARIF shape + level mapping, exit-code matrix, ApiClient errors,
`main()` fail-closed) + 6 workflow/integration (blocked/safe deployments, policy pre-check,
signed verdict, library-finding→SARIF) + 3 signing. Plus a **real subprocess `agentguard scan`
over HTTP** against a live uvicorn (exit 20 + valid SARIF). Full offline suite **179 passed /
6 skipped**; ruff + `mypy --strict` clean (44 files); the `agentguard` CLI runs from the
Docker image; gitleaks clean. No migration this cycle.

## 7. Risks / deferred (honest)

1. **No webhooks / push mode** — pull only.
2. **Signing is symmetric** — integrity, not third-party attestation.
3. **No async** — synchronous scans; fine while runs are seconds.
4. **CLI not published** — the Action's install is overridable until it's on an index.
5. Earlier honest limits (simulation fidelity, library coverage floor, policy override
   loosening, project scope, runtime-declared rules, advisory merge gate) — all unchanged.
